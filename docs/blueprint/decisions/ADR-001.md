# ADR-001: Adopt Project Blueprint and Monorepo Architecture

- Status: Accepted (revised 2026-07-22)
- Date: 2026-06-08
- Owners: Platform team

## Context

Eveider is a pickup locker platform starting from scratch — mobile app (customer + courier), business web portal, admin web dashboard, and a shared parcel lifecycle backend. Registered businesses submit parcels; Eveider handles locker delivery. Without upfront architectural constraints, the codebase risks:

- Duplicated parcel, delivery, and locker logic spread across apps
- Inconsistent role enforcement (customer vs courier vs business vs admin)
- Business data leaking across tenant boundaries (`businessId` scope violations)
- Invalid parcel status transitions implemented ad hoc in multiple places
- No consistent error handling or API contracts between mobile and admin
- Framework code leaking into domain logic (untestable, tightly coupled)
- Authorization gaps from per-route checks that ignore role scope
- Type unsafety from permissive TypeScript usage

A structured blueprint with explicit product context, package boundaries, and operating conventions is needed before any code is written.

## Decision

Adopt the Eveider project blueprint as the single source of architectural truth. This includes:

1. **Monorepo structure** — `apps/mobile-tenant`, `apps/web-manager` + `packages/domain`, `packages/api-contracts`, `packages/data-access`, `packages/ui`, `packages/config-*`
2. **Layered dependency model** — Presentation → Application → Domain → Infrastructure; no upward imports
3. **TypeScript strict mode everywhere** — `any` banned in non-trivial code
4. **Supabase Auth** for session management; role (`customer` | `courier` | `business` | `admin`) enforced on every protected operation; business routes scoped to `businessId`
5. **PostgreSQL via hand-written SQL** (`pg` pool in `packages/data-access`) for relational data; SQL migrations in `db/migrations/`; Supabase for auth and storage where needed
6. **Zod** for all input validation at API boundaries
7. **Shared `ApiResult<T>` error contract** for all async server responses
8. **Parcel lifecycle in `packages/domain`** — canonical statuses and valid transitions; reject invalid transitions in one place
9. **ESLint import rules** to enforce package boundary violations at CI time
10. **`docs/blueprint/templates/project-updates.md`** as first implementation memory source before codebase search
11. **`docs/blueprint/product/`** as authoritative product and domain reference (overview, brand, glossary, feature specs)

### App Responsibilities

| App | Users | Responsibility |
|-----|-------|----------------|
| `apps/mobile-tenant` | Customer, courier | Role-based UI after login; tracking, PIN pickup, scan, drop-off |
| `apps/web-manager` | Admin, business | Unified portal: admin ops dashboard, business registration/onboarding, parcel submission, mobile API host |

Shared business rules (parcel status, locker occupancy, PIN issuance, `businessId` scoping) live in `packages/domain` — not duplicated per app.

### Revision note (2026-07-22)

The original scaffold used three web apps (`web-admin`, `web-business`) and Prisma ORM. These were consolidated without changing product scope:

- **Web consolidation:** `web-admin` + `web-platform` → `apps/web-manager` (admin at `/tableau-de-bord/*`, business at `/entreprise/tableau-de-bord/*`)
- **Mobile rename:** `apps/mobile` → `apps/mobile-tenant`
- **Data layer:** Prisma removed; repositories use `pg` + SQL migrations in `db/migrations/`
- **Client data:** TanStack Query removed from web; Server Components + `src/server/` use-cases for reads

## Consequences

- Positive:
  - Clear ownership boundaries; parcel and locker logic defined once
  - Role scope, business tenancy, and status transitions enforced consistently across all apps
  - Strict TS + Zod catches contract violations at compile time
  - Security by default: auth and role checks always in server code
  - Onboarding is faster with explicit product docs and agent rules
  - Single web deploy surface simplifies mobile API hosting and auth
- Negative:
  - Initial setup overhead for monorepo tooling and ESLint boundary rules
  - Single mobile app must cleanly separate customer and courier modes
  - Requires discipline to keep domain package framework-free
- Neutral:
  - All new features must align with product docs and delivery priority in `AGENT_RULES.md` before implementation

## Alternatives Considered

1. **Separate customer and courier mobile apps** — Clearer separation, but duplicates auth, shared components, and release overhead; rejected for MVP
2. **Plain single Next.js app** — Simpler to start, but no proper mobile surface and weak domain layer; becomes hard to maintain as parcel and locker features grow
3. **Custom ad-hoc structure per feature** — Maximum short-term speed, but leads to inconsistent patterns, duplicated types, and authorization gaps
4. **Keep three separate web apps** — Rejected during 2026-07 refactor; duplicated auth, routing, and API surface without product benefit

## Rollout Plan

- Step 1: Establish `docs/blueprint/product/` and this ADR as Phase 1 foundation documents
- Step 2: Scaffold monorepo folder structure with `packages/*` and apps (Phase 2)
- Step 3: Configure ESLint boundary rules and TypeScript project references (Phase 2)
- Step 4: Implement domain parcel lifecycle and role types in `packages/domain` before feature slices
- Step 5: All subsequent features follow the layered model and delivery priority enforced by CI
- Step 6 (2026-07): Consolidate web apps, migrate data-access to SQL, remove Prisma

## Validation

- ESLint import rule violations fail CI
- TypeScript strict mode errors fail CI
- Product docs under `docs/blueprint/product/` match accepted Eveider definition
- Parcel status transition tests exist for `packages/domain` before business, courier, or admin status-update routes ship
- Business-scoped queries enforce `businessId` — covered by tests on business portal routes
- ADR reviewed and accepted before any scaffolding begins
